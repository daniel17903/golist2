# Device-based list sharing plan

## Recommended stack

### Backend
- **VPS** running **Docker** for simple, low-ops hosting.
- **Fastify (TypeScript)** for the HTTP API and access control logic.
- **Postgres** as the primary document store for list JSON blobs (use `jsonb`).

### Client
- **Device ID**: Generate and store a UUID on first app open (IndexedDB or localStorage).
- **Share tokens**: Random, unguessable tokens that map to a list document.
- **Deletion model**: Use item tombstones (`deleted: true`) instead of hard deletes for safer multi-device sync.

### Why this stack
- Predictable cost with a small VPS and a single database.
- Fits the “no accounts, just device IDs” requirement.
- Simplifies to “list as JSON document” storage while still allowing multi-device access.

---

## Implementation plan

### Phase 1: Foundations
1. **Device identity**
   - Generate a UUID on first launch and persist it.
   - Include `deviceId` on every API request.
2. **List metadata**
   - Add `shareToken`, `createdAt`, and `updatedAt` fields.
   - Keep canonical timestamps generated by the backend.
   - Support multiple share tokens per list; each token can be tracked independently.
3. **Sharing UI**
   - Add a “Share list” action that generates (or reveals) a share link.
   - Add a “Join list” flow that accepts a share link and imports the list.

### Phase 2: API integration
4. **Fastify API**
   - Implement list and item endpoints by `shareToken`.
   - Restrict access via an `Authorization: Bearer <shareToken>` header.
5. **Sync behavior**
   - On load: pull the full remote list JSON and merge into local state.
   - On change: push item-level create/update requests and list name updates.
   - For removals: send item updates with `deleted: true`.

## API spec (v1)

### Authentication
- No user accounts.
- Requests include:
  - `X-Device-Id: <uuid>`
  - `Authorization: Bearer <shareToken>` for protected list access.

### Storage (Postgres)

#### Table: `shared_lists`
```sql
create table shared_lists (
  share_token text primary key,
  list_id uuid not null,
  created_by uuid not null,
  name text not null,
  document jsonb not null,
  created_at timestamptz not null,
  updated_at timestamptz not null
);

create table list_share_tokens (
  token_id uuid primary key,
  list_id uuid not null,
  created_at timestamptz not null,
  redeemed_by uuid[] not null default '{}'
);
```

### Data model

#### List document
```json
{
  "listId": "uuid",
  "shareToken": "random-token",
  "name": "Groceries",
  "items": [
    {
      "id": "uuid",
      "name": "Milk",
      "category": "Dairy",
      "deleted": false,
      "createdAt": "2026-02-12T10:00:00Z",
      "updatedAt": "2026-02-12T10:00:00Z"
    }
  ],
  "createdAt": "2026-02-12T10:00:00Z",
  "updatedAt": "2026-02-12T10:00:00Z",
  "createdBy": "device-uuid",
  "shareTokenIds": ["uuid", "uuid"]
}
```

## Notes
- **Share link security**: treat share tokens like passwords. Use a long random token.
- **Rate limiting**: optional per deviceId to mitigate abuse.
- **Conflict safety**: use `SELECT ... FOR UPDATE` for list mutations and commit merged results atomically.
- **Token redemption**: append the requesting `deviceId` to `redeemed_by` when a token is used by a new device.
